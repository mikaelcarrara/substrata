Governance

Core Principles

  • Tokens as contracts: tokens are stable contracts between design and code; value changes are breaking by default.
  • Single source: the canonical repository is the authorized place to edit tokens.
  • Semantic versioning: follow semver; breaking changes require a major version bump.
  • Traceability: all changes require a PR with justification, changelog entry, and assigned owner.
  • Human review: critical changes require approval from category owners.

Roles and Responsibilities

  • Category Owners
    • Colors - palette and semantic aliases
    • Typography - typographic scales
    • Spacing - spacing scale
    • Motion, Elevation, Opacity, Breakpoints - owners for their respective domains
  • Integration Team - maintains export pipelines (CSS/JSON/TS) and CI jobs
  • Design Team - validates semantic mappings and approves visual identity changes
  • QA Team - monitors visual regressions and compliance metrics

Change Process

  1. Open a PR with description, category, and assigned owner
  2. Run automated validations: schema checks, breaking-change detection, token-aware linting
  3. Review and approval by owners and design team when applicable
  4. Merge and release: publish substrata.css, substrata.json, substrata.d.ts
  5. Deprecation: mark tokens as deprecated in files and changelog; keep compatibility for at least two releases before removal

Quick PR Checklist

  • Description: motivation and impact
  • Category and owner: assigned in the PR
  • Validations: schema and breaking-change checks passing
  • Examples: updated consumption examples (CSS/JSON/TS)
  • Changelog: entry added; deprecation noted if applicable
  • Visual regression: tests approved when required
  • Release notes: prepared for publication

CI and Essential Artifacts

Published Artifacts

  • substrata.css - tokens as CSS Custom Properties
  • substrata.json - tokens in JSON for tools and agents
  • substrata.d.ts - TypeScript types for application consumption

Recommended Jobs

  • build:tokens - generate CSS/JSON/TS from the canonical source
  • lint:tokens - validate schema and detect breaking changes
  • lint:code - detect hardcoded values outside the token whitelist
  • visual-regression - ensure substitutions did not change appearance
  • audit-log - record generations and refactors for review

Design Synchronization

  • Design contract: maintain a design-contract.json mapping tokens to Figma styles (token → value → design reference)
  • Automated export: whenever possible, automate export from the design tool to JSON
  • Periodic validation: compare repository tokens with exported design tokens and report divergences

Success Metrics

  • Adoption: number of repositories consuming substrata-tokens
  • Compliance: percentage of PRs blocked by token-aware linting
  • Rollout time: average time to apply a design change across consumer products
  • Visual regressions: failures in tests after automated refactors
  • Generation accuracy: human acceptance rate of AI-generated outputs

How to Use This Page

  • For teams: follow the checklist before opening PRs; consult owners when in doubt
  • For designers: review design-contract.json and coordinate changes impacting visual identity
  • For engineers: integrate the recommended CI jobs and consume published artifacts
