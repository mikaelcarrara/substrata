Automation

AI & Automation Integration

Substrata can act as a stable design layer in AI-assisted and automated workflows. Its explicit token structure allows automation tools and AI systems to read, validate, and generate UI code without coupling to any specific framework.

AI-assisted workflows

Design tokens in Substrata are explicit, semantic, and framework-agnostic. This makes them suitable as inputs and constraints for automation and AI systems.

Rather than generating arbitrary styles, AI operates within a bounded design space defined by tokens.

Example use cases

  • Generate UI components from design prompts constrained by tokens
  • Automatically validate color, spacing, and typography consistency
  • Refactor UI implementations across frameworks while preserving design intent
  • Build design-to-code pipelines where tokens act as constraints, not suggestions.

Why tokens matter for AI

  • Tokens reduce ambiguity
  • Subjective design decisions become explicit, machine-readable constraints
  • Automation becomes predictable, auditable, and safe at scale

Example prompts using tokens as input

1. Generate a component constrained by tokens

Generate a Card component using only the following design tokens:

- Colors: neutral-0, neutral-200, neutral-900
- Spacing: space-3, space-4, space-5
- Radius: radius-md
- Typography: font-size-md, font-weight-medium

Do not introduce new colors, spacing values, or font sizes.
Output plain HTML and CSS.

Enforces

  • No arbitrary values
  • No framework assumptions
  • Tokens act as hard constraints

2. Validate a component against tokens

Analyze the following CSS and report any values that are not mapped to the provided tokens:

- Allowed spacing tokens: space-1 to space-6
- Allowed colors: neutral scale and brand scale
- Allowed font sizes: font-size-sm, font-size-md, font-size-lg

Return a list of violations and suggested token replacements.

Use case

  • Automated design linting
  • CI validation
  • Consistency checks at scale

3. Refactor a component across frameworks

Refactor this React component to plain HTML and CSS.

Preserve all design decisions by replacing hardcoded values with the following tokens:
- Colors
- Spacing
- Typography
- Border radius

Do not change the visual output.
Do not introduce framework-specific patterns.

Outcome

  • Same design
  • Different implementation
  • Tokens remain the invariant layer

4. Design-to-code from intent, not pixels

Create a form layout based on these constraints:

- Use Substrata form tokens and base styles
- Inputs must follow spacing and typography tokens
- Focus states must use brand tokens
- No custom values allowed

The result should be accessible and framework-agnostic.

Why this works

  • The prompt describes rules, not visuals
  • Tokens replace subjective interpretation

5. Token-driven variation generation

Generate three visual variants of the same component.

The only allowed difference between variants is:
- brand-300
- brand-500
- brand-700

All other tokens must remain unchanged.

Use case

  • Theming
  • A/B testing
  • Brand adaptation without redesign

AI Prompt Templates (Token-driven)

These templates can be reused in automation pipelines, CI workflows, or AI agents.

Component Generation Template

Purpose:
Generate UI components strictly constrained by design tokens. 

ROLE:
You are a UI code generator constrained by design tokens.
 
INPUT:
- Component type: {{component_name}}
- Allowed tokens:
- Colors: {{color_tokens}}
- Spacing: {{spacing_tokens}}
- Typography: {{typography_tokens}}
- Radius: {{radius_tokens}}

RULES:
- Do not introduce values outside the provided tokens.
- Do not assume any framework.
- Output plain HTML and CSS only.

OUTPUT:
- Semantic HTML
- Minimal, token-based CSS

Token validation / linting template

Purpose:
Detect design inconsistencies automatically. 

ROLE:
You are a design token validator.

INPUT:
- Source code: {{code_input}}
- Allowed tokens:
- Colors: {{color_tokens}}
- Spacing: {{spacing_tokens}}
- Typography: {{typography_tokens}}

TASK:
- Identify any hardcoded values not mapped to tokens.
- Suggest the closest valid token replacement.

OUTPUT:
- List of violations
- Line reference (if available)
- Suggested token

Cross-Framework Refactor Template

Purpose:
Preserve design decisions while changing implementation. 

ROLE:
You are a refactoring agent.

INPUT:
- Source component: {{source_code}}
- Target output: {{target_format}}
- Design tokens: {{token_set}}

RULES:
- Preserve visual output.
- Replace all hardcoded values with tokens.
- Do not introduce framework-specific abstractions.

OUTPUT:
- Refactored component using only tokens

Design-to-Code Constraint Template

Purpose:
Generate UI from intent, not from visual guesswork. 

ROLE:
You are a design-to-code generator.

INPUT:
- UI intent: {{intent_description}}
- Token system: {{token_set}}

CONSTRAINTS:
- Use only provided tokens
- Respect accessibility best practices
- No custom values

OUTPUT:
- Token-compliant UI markup

Controlled Variation Template

Purpose:
Create safe visual variations without redesign. 

ROLE:
You are a variation generator.

INPUT:
- Base component: {{component_code}}
- Variable tokens: {{variable_tokens}}

RULES:
- Only variable tokens may change.
- All other tokens must remain identical.

OUTPUT:
- Variant A
- Variant B
- Variant C

How this fits an AI pipeline

  • Design tokens define the contract
  • Prompts act as executable specifications
  • AI operates as a constrained executor, not a decision-maker

This enables Substrata to integrate with:

  • CI validation
  • Design QA automation
  • Design-to-code agents
  • Multi-framework refactors
  • AI-assisted theming
